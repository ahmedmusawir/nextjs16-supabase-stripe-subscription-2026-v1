# Changelog — v0.2.0
**Date:** 2026-04-12
**Type:** Feature Release

---

## Summary

This release implements the **v2 Superadmin Portal** — a full user management dashboard accessible only to superadmin-role users. It introduces the `profiles` table, a smart database trigger, corrected RLS policies, and a complete CRUD UI with pagination, forms, and toast notifications.

---

## New Features

### Superadmin Portal UI
- **Dashboard** (`/superadmin-portal`) — paginated 3-column card grid showing all users with name, email, role badge, and Edit/Delete actions. 6 users per page, URL-based pagination (`?page=N`).
- **Add User** (`/superadmin-portal/add-user`) — form with Name, Email, Password, and Role dropdown (`admin` / `member`). Inline spinner on submit, success/error toast.
- **Edit User** (`/superadmin-portal/edit/[id]`) — form with editable Name and Role, read-only Email. Inline spinner on submit, success/error toast.
- **Delete User** — per-card delete button with browser confirm dialog, inline spinner, success/error toast.
- **SuperadminSidebar** — sidebar navigation with Dashboard and Add User links, mirroring the members/admin portal sidebar pattern.

### Server Actions (`actions.ts`)
- `getUsers(page)` — paginated fetch using two-query merge pattern (profiles + user_roles)
- `getUserById(id)` — parallel fetch via `Promise.all()`, merged in JS
- `addUser(formData)` — creates user via `auth.admin.createUser()` with `full_name` and `role` in metadata; trigger handles all DB inserts automatically
- `editUser(userId, formData)` — updates name in auth metadata and role in `user_roles`
- `deleteUser(userId)` — deletes auth user; cascade removes `profiles` and `user_roles` rows

### New UI Components
- `SuperadminSidebar.tsx` — sidebar for the superadmin route group
- `DeleteUserButton.tsx` — client component with confirm + spinner + toast
- `AddUserForm.tsx` — React Hook Form + Zod, role dropdown, spinner
- `EditUserForm.tsx` — React Hook Form + Zod, read-only email, role dropdown, spinner
- `src/components/ui/select.tsx` — shadcn-style Radix UI Select component (was missing from project)

## Database Changes

### New Table: `public.profiles`
Stores public-facing user profile data (name, email). Synced automatically by trigger on every new auth user creation.

```sql
CREATE TABLE public.profiles (
  id         uuid PRIMARY KEY REFERENCES auth.users (id) ON DELETE CASCADE,
  full_name  text,
  email      text,
  created_at timestamp with time zone DEFAULT now()
);
```

### Updated Trigger: `handle_new_user()` (Mark IV)
Replaces the previous single-table trigger. Now:
- Reads `role` from `raw_user_meta_data` — if present, assigns it; otherwise defaults to `'member'`
- Reads `full_name` from `raw_user_meta_data`
- Inserts into **both** `user_roles` AND `profiles` atomically on every new auth user

### New RLS Policies on `profiles`
- Users can read and update their own profile row
- Superadmins can read and update **all** profile rows (required for the dashboard to function)

### Migration File
`docs/migration_add_profiles.sql` — safe to run on existing databases with live `user_roles` data. Includes backfill for existing users.

## Notes

- The `profiles` and `user_roles` tables are sibling tables — both FK to `auth.users` but with no direct FK between them. PostgREST nested selects do not work across sibling tables; always use the two-query merge pattern.
- The smart trigger is the single point of truth for `profiles` and `user_roles` row creation. Never manually insert into these tables when creating users via `auth.admin.createUser()`.
- The first superadmin must still be promoted manually via SQL (one-time operation). See `docs/DATABASE_SETUP.md` Step 5.
