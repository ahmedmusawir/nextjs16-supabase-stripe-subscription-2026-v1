# Changelog — v0.3.0
**Date:** 2026-04-12
**Type:** Minor — Admin Portal Feature (user management for admin role)

---

## Summary

Complete Admin Portal implementation for admin role users. Admins can view and manage admin + member users with restricted permissions compared to superadmins. No database changes. No new dependencies.

---

## New Features

### 1. Admin Portal — User Management Dashboard
**Route:** `/admin-portal`
**Files:** `(admin)/admin-portal/page.tsx`, `AdminPortalPageContent.tsx`, `actions.ts`

- Paginated user list showing admin + member role users (superadmins filtered out)
- Card-based grid layout identical to Superadmin Portal
- Role displayed with color coding: `Role: Admin` (red), `Role: Member` (green)
- **Conditional delete button:**
  - Admin role users: **Edit button only** (no delete)
  - Member role users: **Edit + Delete buttons**
- Pagination with Previous/Next buttons
- Uses `createAdminClient()` (service role) for all queries to bypass RLS

### 2. Admin Portal — Edit User Form
**Route:** `/admin-portal/edit/[id]`
**Files:** `edit/[id]/page.tsx`, `edit/[id]/EditUserForm.tsx`

- **Only name field is editable** — admins cannot change roles or passwords
- Email displayed as read-only with helper text: *"Email cannot be changed"*
- Role displayed as static colored text with helper text: *"Role cannot be changed by admins"*
- Form validation with Zod (name required)
- Red error messages (`text-red-500 dark:text-red-400`)
- Success redirects to `/admin-portal`
- "Back to Dashboard" button with arrow icon

### 3. Admin Portal — Delete User Action
**File:** `DeleteUserButton.tsx`
- Shadcn Dialog modal for delete confirmation (same as Superadmin Portal)
- Red bordered Delete button with hover states
- Only renders for member role users
- Calls `deleteUser()` server action which uses `adminClient.auth.admin.deleteUser()`
- Cascades to `profiles` and `user_roles` via FK constraints

### 4. Admin Portal — Server Actions
**File:** `(admin)/admin-portal/actions.ts`

**`getUsers(page)`**
- Fetches paginated profiles + user_roles (two-query merge pattern)
- Returns admin + member users only (superadmins filtered in component)
- Uses `createAdminClient()` to bypass RLS

**`getUserById(userId)`**
- Fetches single user's profile + role for edit form
- Uses `createAdminClient()` to bypass RLS

**`editUser(userId, { name })`**
- Updates `auth.users.user_metadata.full_name` and `profiles.full_name`
- **Does not update role** — admins cannot change roles
- Revalidates `/admin-portal` path

**`deleteUser(userId)`**
- Deletes user from `auth.users` via admin API
- Cascades to `profiles` and `user_roles` automatically
- Only callable for member users (enforced in UI, not server action)

### 5. Admin Sidebar Update
**File:** `components/layout/AdminSidebar.tsx`
- Removed old booking-related menu items (New Booking, Bookings)
- Simplified to:
  - **Suggestions:** Dashboard (links to `/admin-portal`)
  - **Settings:** Profile (links to `/profile`)

### 6. Admin Portal Loading State
**File:** `(admin)/loading.tsx`
- Updated from `min-h-screen` to `min-h-[60vh]` to scope spinner to content area
- Navbar and sidebar remain visible during page transitions

## Architecture Notes

### Permission Model
- **Superadmin Portal:** Full CRUD on all users (create, read, update role/name, delete)
- **Admin Portal:** Restricted CRUD
  - **Create:** ❌ No add user capability
  - **Read:** ✅ View admin + member users (no superadmins)
  - **Update:** ✅ Edit name only (no role changes, no password)
  - **Delete:** ✅ Delete member users only (admin users protected)

### Security
- All server actions use `createAdminClient()` (service role key) to bypass RLS
- Route protection via `protectPage([AppRole.ADMIN])` in `(admin)/layout.tsx`
- UI-level permission enforcement (delete button conditional rendering)
- Server actions do not enforce role-based restrictions — relies on route-level protection

### UI/UX Consistency
- Identical card grid layout to Superadmin Portal
- Same color-coded role labels (red=admin, green=member)
- Same form styling, error message colors, button styles
- Same pagination pattern
- Same loading spinner behavior (scoped to content area)

## Known Limitations

- **No server-side role enforcement in delete action** — `deleteUser()` does not check if the target user is an admin. This is acceptable since the delete button is hidden in the UI for admin users, and the route is protected by `protectPage([AppRole.ADMIN])`. If direct API calls are a concern, add role check in `deleteUser()`.
- **No add user capability** — admins cannot create new users. Only superadmins can add users via the Superadmin Portal.
