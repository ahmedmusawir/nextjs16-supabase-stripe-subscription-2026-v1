# Changelog — v0.2.1
**Date:** 2026-04-12
**Type:** Patch — UI/UX Fixes (found during manual testing of v0.2.0)

---

## Summary

Six UI/UX issues found during the first manual testing pass of the Superadmin Portal were fixed. No database changes. No new dependencies.

---

## Bug Fixes

### 1. Delete button — red thick border styling
**File:** `SuperadminPortalPageContent.tsx`, `DeleteUserButton.tsx`
- Delete button was using the filled `destructive` variant with no border, inconsistent with the Edit button's outlined style
- Changed to `variant="outline"` with explicit `border-2 border-red-500 text-red-500` and dark mode equivalents
- Edit button also given explicit `border-2 border-slate-400` to match the thick-border intent

### 2. Superadmin users hidden from dashboard list
**File:** `SuperadminPortalPageContent.tsx`
- Superadmin accounts were appearing in the user management list, which is incorrect — superadmins manage others, not themselves
- Added a client-side filter: `users.filter((u) => u.role !== "superadmin")` after fetching

### 3. Delete confirmation — replaced browser `confirm()` with shadcn Dialog modal
**File:** `DeleteUserButton.tsx`
- The native browser `confirm()` dialog is unstyled, blocks the thread, and looks out of place in a polished UI
- Replaced with a proper shadcn `Dialog` component with Cancel and "Yes, Delete" buttons, spinner state on the confirm button, and displays the user's name or email in the prompt
- `userEmail` prop added to `DeleteUserButtonProps` to support showing email as fallback when name is null

### 4. Role dropdown — transparent background (unreadable text)
**Files:** `AddUserForm.tsx`, `EditUserForm.tsx`
- The `SelectContent` dropdown was rendering with a transparent background, making the option text invisible against the page background
- Fixed by adding explicit `bg-white dark:bg-slate-800` with border to `SelectContent` in both forms

### 5. Submit buttons — disappearing on hover
**Files:** `AddUserForm.tsx`, `EditUserForm.tsx`
- The `bg-slate-500` + `bg-slate-800` combo had no `hover:` class, causing the button to visually vanish on hover (the inherited hover was overriding with transparent)
- Fixed by changing to `bg-slate-700 hover:bg-slate-900 dark:bg-slate-700 dark:hover:bg-slate-600` for explicit, visible hover states in both modes

### 6. Global loading spinner covering full screen during page navigation
**File:** `src/app/(superadmin)/loading.tsx`
- The `loading.tsx` file used `min-h-screen` with a full white/dark background, completely covering the navbar and sidebar on every route transition inside the superadmin portal
- This pattern is correct for login/logout transitions but wrong for in-app navigation — the layout chrome (navbar, sidebar) must always remain visible
- Fixed by changing to `min-h-[60vh]` with no background override, so the spinner renders inside the content area only

## Notes

- The `loading.tsx` approach (Next.js streaming loading UI) fires on any server component fetch inside the route group. For login/logout, a full-screen overlay is appropriate. For in-app navigation it is not — always scope `min-h` to the content area, never `min-h-screen` inside a layout that has persistent chrome.
- The `confirm()` → `Dialog` upgrade applies to any delete/destructive action in the project going forward. Native browser dialogs should be avoided in production UIs.
