## File Header (Use At Initialization)

This is the seed content for a fresh `SECURITY_FINDINGS.md` file. Generated when `workflow/00-initialize-folder.md` runs and the file doesn't exist.

```markdown
# Security Findings — <PROJECT_NAME>

> **Purpose:** Track real security issues in this project's code (not third-party dep vulnerabilities — those are in the audit cycle). Includes: PII / financial data leaks, missing input validation, error message leaks, auth gaps, fraud vectors.
>
> **How entries get added:** Use `stark-repo-security/repo-security-findings-tracker/` skill. The skill walks the capture conversationally.
>
> **How entries get read:** Per the project root `CLAUDE.md` reading reflexes (if configured) — at session start, at phase close, at pre-deploy, and during code review.

## Status Legend

- 🔴 **OPEN** — Confirmed, no fix yet
- 🟡 **IN PROGRESS** — Fix planned or in flight
- ✅ **FIXED** — Addressed and verified
- ⚪ **DEFERRED** — Acknowledged, intentionally not yet addressed
- ⚫ **WON'T FIX** — Documented decision not to address (rationale below)

## Severity Legend

- 🔴 **CRITICAL** — Production data exposure, RCE, auth bypass, payment fraud vector
- 🟠 **HIGH** — PII / financial data leak, missing input validation on money/PHI paths
- 🟡 **MEDIUM** — Hardening opportunity, missing defense-in-depth, fraud-enabling without active leak
- 🟢 **LOW** — Code smell with security implication, best-practice deviation
```

## Worked Example (For Reference)

Based on Dockbloxx Finding #2 (the canonical missing-input-validation case):

```markdown
## Finding #2 — Missing Input Validation on /api/create-payment-intent

**Status:** 🔴 OPEN
**Severity:** 🟠 HIGH (payment fraud vector, money path)

### Problem

The `/api/create-payment-intent` POST handler reads `amount` and `currency` from the request body and passes them directly to Stripe's `paymentIntents.create()` without schema validation. An attacker can send any number — including negative values, NaN, or values that bypass the cart's calculated total — and Stripe will create the PaymentIntent with whatever value is supplied. Combined with the fact that the client-side cart total isn't re-verified server-side, this creates a fraud vector where an attacker could pay $0.01 for a $100 order.

### Discovery

Noticed while grepping for axios usage during the 2026-05-17 security audit (the route handler doesn't use axios — caught the missing validation while reading the file).

### Recommended Fix

Add a zod schema at the top of the POST handler:
1. Define schema enforcing `amount: z.number().positive().int()` and `currency: z.enum(['usd', 'eur', ...])`
2. Parse `await request.json()` through the schema
3. Re-verify the amount against the server-side cart total before calling Stripe
4. Return 400 on validation failure with a generic message (no schema details in response)

### Mitigation Until Fix

None currently active. Consider rate-limiting the endpoint at the WAF/CDN layer until the fix lands.

### Files

- `src/app/api/create-payment-intent/route.ts` (POST handler, lines 12-45)

---
```

## Status Update Pattern (Path C)

When a finding's status changes, the surgical edit is:

1. Replace the `**Status:**` line with the new state
2. Append a `### Resolution / Decision Record` section before the closing `---`

### FIXED Update Example

Replace:
```markdown
**Status:** 🔴 OPEN
```

With:
```markdown
**Status:** ✅ FIXED 2026-05-19
```

Append before `---`:
```markdown
### Resolution

Added zod validation schema to POST handler. Schema enforces positive integer amount and currency enum. Server-side re-verification against cart total now precedes Stripe call. Tests added in `tests/api/create-payment-intent.test.ts` to lock the validation in. Commit: abc1234.

```

### DEFERRED Update Example

Replace:
```markdown
**Status:** 🔴 OPEN
```

With:
```markdown
**Status:** ⚪ DEFERRED 2026-05-19
```

Append before `---`:
```markdown
### Decision Record

Deferred to Phase 8 (HIPAA hardening pass). Rationale: requires architectural decision on validation framework (zod vs valibot) that's scoped to the broader hardening effort. Review date: 2026-06-15. Un-defers if: a payment fraud incident occurs OR validation framework decision is made earlier.

```

### WON'T FIX Update Example

Replace:
```markdown
**Status:** 🔴 OPEN
```

With:
```markdown
**Status:** ⚫ WON'T FIX 2026-05-19
```

Append before `---`:
```markdown
### Decision Record

Accepted risk. Rationale: this endpoint is internal-only, protected by VPN + auth, never exposed to public internet. Threat model doesn't include authenticated insiders for this surface. Signed off by: Tony (operator). If endpoint ever becomes publicly accessible: this decision is voided and the finding re-opens automatically.

```

## Numbering Rules

- Findings get sequential numbers in order of discovery
- Numbers are NEVER reused — if Finding #3 is later WON'T FIX, #4 still stays #4
- To determine the next number, read the file and find the highest existing `## Finding #N` heading; new entry is N+1
- If file has no findings yet, first entry is `## Finding #1`

Stable numbering means external references (CHANGELOG entries, commit messages, code comments) remain valid forever.

## What Goes In This File vs CLEANUP_BACKLOG.md

| This File (Findings) | CLEANUP_BACKLOG.md |
|---|---|
| Real security issues | Hygiene with no security angle |
| Blocks deploy if HIGH/CRITICAL + OPEN | Doesn't block anything |
| Severity ranked (CRITICAL/HIGH/MEDIUM/LOW) | Risk-if-left ranked (HIGH/MEDIUM/LOW/VERY LOW) |
| Status lifecycle (OPEN → FIXED) | Open/done; lighter ceremony |
| Numbered sequentially | Tracked by title |
| Read at every session start, phase close, pre-deploy | Reviewed periodically |

When uncertain, walk `decision-trees/finding-vs-cleanup.md`.
